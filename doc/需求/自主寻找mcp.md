# 自主寻找 MCP（极简版）需求说明

## 1. 背景

当前 AgentDSL 中，Agent 只能使用预先配置好的工具。  
当用户请求超出当前工具能力时，流程会失败或需要人工补配置。

目标是增加一个低成本能力：当工具不存在时，Agent 可以自动去 MCP 仓库发现并挂载合适工具，然后继续执行任务。

本需求强调两点：
- 用户使用要简单
- 技术实现要低复杂度、低成本

## 2. 目标与非目标

### 2.1 目标

- 在 `ToolNotFound` 场景下自动触发 MCP 发现与挂载。
- 提供最少 DSL 配置，开箱可用。
- 仅实现一条可跑通、可观测、可回退的兜底链路。

### 2.2 非目标（V1 不做）

- 多仓库聚合搜索
- 复杂评分模型与自动优选
- 人工审批 UI 流程
- 容器/微虚拟机级隔离
- 自动升级、版本治理、灰度发布

## 3. DSL 设计（极简）

```groovy
agent "助手" {
    auto_discover_mcp true
    mcp_registry "mcp.so"
}
```

说明：
- `auto_discover_mcp`：开启后，仅在工具缺失时触发自动发现。
- `mcp_registry`：MCP 检索来源。V1 仅支持单一内置源，默认 `mcp.so`。

## 4. 运行流程（V1）

### 4.1 触发条件

- Agent 尝试调用工具失败，错误类型为 `ToolNotFoundException`（或等价错误码）。

### 4.2 执行步骤

1. 根据当前任务与缺失工具名构造搜索词。
2. 调用 registry 搜索候选 MCP 服务（取前 3 个候选）。
3. 用简单规则选出 1 个候选：
   - 优先有标准启动命令（如 `npx -y @modelcontextprotocol/...`）
   - 优先依赖最少、启动参数最少
4. 使用 `ProcessBuilder` 启动 MCP 进程。
5. 发起 `tools/list` 握手，注册可用工具到当前 Agent 上下文。
6. 重试原始工具调用一次：
   - 成功：继续任务
   - 失败：返回可读错误，并提示手动配置

### 4.3 失败回退

- 搜索无结果：直接回退原错误，并给出“建议手动配置工具”提示。
- 启动失败：记录日志并回退。
- 握手失败：终止该 MCP 进程并回退。

## 5. 安全与稳定性（低成本最小集）

V1 不追求重型隔离，但必须有以下硬限制：

- **命令白名单**：仅允许预定义前缀（如 `npx -y @modelcontextprotocol/`）。
- **超时控制**：启动超时、调用超时（例如 30 秒）。
- **资源控制**：单任务动态挂载次数上限（例如 1 次），避免无限重试。
- **审计日志**：记录搜索词、候选、最终选择、启动命令、结果状态。

## 6. 实现边界与成本评估

### 6.1 主要改动点

- Agent 执行器：增加 `ToolNotFound` 兜底分支
- MCP Client 管理器：支持运行时新增连接与工具注册
- 配置解析：增加 2 个 DSL 字段
- 日志模块：新增动态发现事件日志

### 6.2 复杂度评估

- 研发复杂度：中低
- 技术风险：中（主要在第三方 MCP 启动稳定性）
- 可交付节奏：建议 1 到 2 周完成 V1

## 7. 验收标准（V1）

- 开启 `auto_discover_mcp` 后，遇到缺失工具时能自动触发发现流程。
- 至少 1 个公开 MCP 场景可稳定跑通（例如天气或 GitHub 查询类）。
- 失败场景可正确回退，不影响主流程稳定性。
- 审计日志完整可查。
- 关闭 `auto_discover_mcp` 时，行为与当前版本一致。

## 8. 后续迭代建议（V1.1+）

- 本地缓存已成功挂载的 MCP，减少重复搜索与冷启动。
- 增加 registry 备选源（最多 2 个）。
- 增加可配置风险级别（strict/balanced），但保持 DSL 简洁。
